# V2 Subscription API Documentation

## Overview

The V2 API introduces **token versioning** to support the new app version while maintaining backward compatibility with the old app. 

**Key Point:** V1 tokens only work with V1 endpoints, and V2 tokens only work with V2 endpoints. This ensures complete isolation between the old and new app versions.

---

## Token Versions

### V1 Tokens (Old App)
- Generated by: `POST /admin/tokens` or `POST /admin/tokens/v1`
- Used by: `/app/activate` endpoint
- App: Old app version
- Format: Same as before (6 letters + 2 numbers)

### V2 Tokens (New App)
- Generated by: `POST /admin/tokens/v2`
- Used by: `/app/v2/activate` endpoint
- App: New app version
- Format: Same as V1 (6 letters + 2 numbers)

## Admin Endpoints

### Creating V1 Tokens (Old App)

**Endpoint:** `POST /admin/tokens/v1`

**Headers:**
```
X-API-KEY: your-admin-api-key
Content-Type: application/json
```

**Query Parameters:**
- `days` (int, required): Token validity in days
- `count` (int, required): Number of tokens to generate

**Example Request:**
```bash
curl -X POST "http://localhost:8000/admin/tokens/v1?days=30&count=5" \
  -H "X-API-KEY: 1b4f4fcb-ab34-4963-a762-6d80591a9fd6"
```

**Response:**
```json
{
  "tokens": ["ABCDEF01", "GHIJKL02", "MNOPQR03", "STUVWX04", "YZABCD05"],
  "version": "v1"
}
```

## Device Limit Rules

Both V1 and V2 activation endpoints follow the same device limit rules:

- **Max Devices per Token:** 2 devices
- **Max Devices per Email:** 2 devices
- **Device Activation Behavior:**
  - First activation: Device expires at token's expiry timestamp
  - Re-activation on same device with same token: Returns current expiration (no extension)
  - Activation on new device with same token: Extends by token duration days

---

## Token Behavior Specification

### Same Device, Same Token
- If device is already active with this token: Returns current `expires_at` without changes
- Example:
  ```json
  Request: { "mac": "AA:BB:CC:DD:EE:FF", "token": "EFGHIJ06", "email": "user@example.com" }
  Response: { "expires_at": "2026-07-24T12:00:00+00:00" }
  (No extension, returns same expiry)
  ```

### New Device, Same Token
- Device limit must not be exceeded (max 2 devices per token)
- New expiration: `now + token.duration_days`
- Example:
  ```json
  Request: { "mac": "AA:BB:CC:DD:EE:00", "token": "EFGHIJ06", "email": "user@example.com" }
  Response: { "expires_at": "2026-07-24T12:00:00+00:00" }
  (New device added, new expiry calculated)
  ```

### Subscription Extension
- When user redeems a **different** token on same device: Device expiration extended
- New expiration: `max(now, current_device_expires_at) + new_token.duration_days`

## Migration Guide

### For Frontend Developers

#### Step 1: Check Which App Version You're Building
- **Old App (V1):** Continue using `/app/activate` with V1 tokens
- **New App (V2):** Use `/app/v2/activate` with V2 tokens

#### Step 2: Update Token Input
When user enters a token, validate it works with your endpoint:
```javascript
// New App Example
async function activateToken(mac, email, token) {
  const response = await fetch('/app/v2/activate', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      mac_address: mac,
      email: email,
      token: token
    })
  });
  
  if (response.status === 400) {
    const error = await response.json();
    if (error.detail.includes('version mismatch')) {
      // User entered wrong token version
      showError('This token is not compatible with your app version');
    }
  }
  
  return response.json();
}
```

#### Step 3: Error Handling
Handle the new error messages:
- `"Token version mismatch"` → User has wrong token version
- `"Invalid token"` → Token doesn't exist
- `"Token expired"` → Token's validity period has passed
- `"Token limit reached"` → Token already used on 2 devices

#### Step 4: Testing
```bash
# Test V1 token with V2 endpoint (should fail)
curl -X POST "http://localhost:8000/app/v2/activate" \
  -H "Content-Type: application/json" \
  -d '{"mac_address":"AA:BB:CC:DD:EE:FF","email":"test@example.com","token":"ABCDEF01"}'

# Response: "Token version mismatch - this endpoint requires V2 tokens"

# Test V2 token with V2 endpoint (should succeed)
curl -X POST "http://localhost:8000/app/v2/activate" \
  -H "Content-Type: application/json" \
  -d '{"mac_address":"AA:BB:CC:DD:EE:FF","email":"test@example.com","token":"EFGHIJ06"}'

# Response: {"expires_at": "2026-07-24T12:00:00+00:00"}
```

## API Comparison: V1 vs V2

| Feature | V1 Endpoint | V2 Endpoint |
|---------|-----------|-----------|
| **Path** | `/app/activate` | `/app/v2/activate` |
| **Token Version** | V1 only | V2 only |
| **Token Format** | 6 letters + 2 numbers | 6 letters + 2 numbers |
| **Device Limit** | 2 per token | 2 per token |
| **Response Format** | `{"expires_at": "..."}` | `{"expires_at": "..."}` |
| **Error Handling** | Same | Same + version mismatch check |
| **Backward Compatible** | Yes | No (new app only) |

## FAQ

### Q: Can I activate a V1 token on the new app?
**A:** No. V1 tokens only work with `/app/activate` (old app). You'll get error: `"Token version mismatch - this endpoint requires V2 tokens"`

### Q: Can I activate a V2 token on the old app?
**A:** No. V2 tokens only work with `/app/v2/activate` (new app). The old app endpoint `/app/activate` will reject V2 tokens with the same error.

### Q: What if I activate the same V2 token on two different devices?
**A:** Both activations succeed. Each device gets its own device record. Third device will fail with `"Token limit reached"`.

### Q: Can a user migrate from old app to new app?
**A:** Yes, but they need a new V2 token. If they had V1 token on old app, they must get a V2 token for the new app.

### Q: How long do tokens last?
**A:** Tokens have a validity period set by `duration_days` (configured when token is created). If the token expires, it cannot be used for new activations. Existing devices are not affected.

### Q: What happens if I try to activate an already-used token on the same device?
**A:** The endpoint returns the current `expires_at` without extending it. No changes are made.

## Rollback Instructions

If you need to roll back:

1. Stop accepting V2 tokens in your frontend
2. Continue using `/app/activate` for all users
3. Old V1 tokens continue to work indefinitely
4. Database migration is reversible (see alembic downgrade docs)
